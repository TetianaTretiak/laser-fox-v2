Overview

Clinical trials produce data. If that data could be accessed through a consistent API, it could be used to power various tools and analyses of clinical trial data. As it is today, pretty much every study stores their data in different formats, which makes analysis ad hoc.

SDW stands for Study Data Warehouse. Its goal is to take all of the varied data from clinical trials and load it into one consistent database. Basically, it is an ETL (extract-transform-load) system.

Concepts

In SDW, “mapping” refers to transformations applied to convert input data into the desired output format. By default, SDW does nothing. All output must be explicitly mapped by creating output domains. In SDW terminology, this means that for each input domain (table of raw data) you want in your output, you need to create an output domain (table of transformed data) and then include maps associated with each output domain to explicitly define what data and transformations you want.

Input Domains

Each input domain is a table of data. For data coming from an inherently tabular source, such as a CSV file, each file will produce one input domain. Some other data sources, such as ODM, contain many different types of data, resulting in multiple input domains.

The Input Domains screen shows all of the input domains associated with a study.

Output Domains

On the Input Domains screen, click the “Map” button next to an input domain, for instance AE3001_LF1.igAE3001b_F1_S2.igAE3001b_F1_S2. This takes you to the main mapping interface.

The screen is divided into four sections. For now, just look at the top left, Output Domains. Each input domain can have any number of output domains.

You can create a new output domain by clicking the [+ NEW] button. This opens a form with three fields: Name, Category, and Output Domain. These fields are very confusingly named, like many things in SDW:

  • Name and Category: treat these as just one field, always put the same value in both. When someone requests the mapped output from SDW, they have to specify the name of the output domain that they want. So for example, if you had a study mapped in two different ways for two different applications, you could have one set of output domains called “my-mapping-1” and another called “my-mapping-2”. Those would be completely separate, but you would want to stay consistent in all your related output domains - every single one should have the same name and same category. You may see some called “mapping-utils”. That is the name that the programmatic API uses (see Programmatic API below), so please don’t use that name for your output domains.

  • Output Domain: The SDTM-ish name of the output you are creating. If you’re mapping adverse events data, call the output domain “AE”. I say “SDTM-ish” because we are often asked to map things in a non-SDTM way, in which case you can just make up your own name, but it must be all capital letters, and it should be at least relatively short (it doesn’t have to be 2 characters like SDTM, but don’t make it 20 characters long, that would look horrible).

Based on those descriptions, it is hopefully clear that each input domain can have between 0 and N output domains and that output domains with different names are considered completely separate. The interesting thing is, you can still have between 0 and N output domains with the same name. Here are the scenarios:

  • If an input domain has no output domains with a given name, then no output will be created from this input domain. However, the data from this input domain could still appear somewhere in the output, if the data is joined in to an output domain associated with another input domain (see Join Transform below). This is common, because often there will be EDC forms that nobody cares about viewing in CMR, in which case we don’t need to waste time mapping them.

  • If an input domain has one output domain with a given name, then each row from the input domain will be transformed into a row of output, based on the transformations defined in the maps. This is also a common scenario, like if a form called “AE_DATA” is filled out every time an adverse event occurs, we will want to map that to an output domain called “AE”.

  • If an input domain has more than one output domain with a given name, then multiple outputs will be created from one input. This is rarely used. One example of usage: Lilly sends lab and electrocardiogram data in the same file, and we want to map those to LB and EG output domains, respectively. So we create two output domains and use a filter to include only lab or electrocardiogram data in the appropriate domain (see Filtering below). Another example of multiple output domains is the disposition output domain DS, which includes rows representing critical events in the progression of a subject through a trial (like “Screened”, “Randomized”, “Completed”, “Withdrawn”, etc.). This often comes from multiple different input domains, and sometimes some of those input domains are used for other things. Like a demographics form may be used to create the DM domain, but might also be used to create a DS row indicating that the subject completed screening.

The practical result of all this is that when mapping a study, make the name and category the same for all the related output domains, and use some unique name so you don’t step on anyone’s toes.

Mapping

With input domains and output domains out of the way, now let’s discuss the core of mapping. Maps run on records of data, like a row in a spreadsheet or database. Each record (called a “raw” record) in and input domain is converted into a record in the output domain (called a “transformed” record).

A map is an individual transformation, taking either the raw or transformed record as input, and producing some modification to the transformed record.

Maps run sequentially, in the same order they are displayed in the list of maps on the bottom left of the mapping screen. This is important, because sometimes you will have a map that depends on the result of another map, in which case proper order matters.

When mapping, it is often useful to look at the data. Raw data records can be viewed by clicking “Fetch” in the top right. Then below that, transformed records can be viewed by clicking “Apply Maps”. If any errors are produced during mapping, they will be displayed in the “Mapping Errors” table.

Types of Maps

To create a map, select the type of map from the dropdown box and press [+ New]. In the sections below

Rename Transform

The most common type of map is a rename transform. As described above, by default SDW does nothing. Everything you want in the transformed record must be explicitly defined. So if there is a column named X in the raw record and you want it to show up in the transformed record as a column named X, you need to create a rename transform to do that.

When you create a new rename map, you need to supply several pieces of information:

  • Name: Can by anything. This is just the name of the map that you will see in the list of maps

  • Mapset Name: Like a category or tag. This is not used by anything in the actual mapping process, but you can use it when importing maps (see Don’t Repeat Yourself below).

  • Enabled: All maps can be enabled/disabled by clicking this checkbox or by clicking the Enable/Disable button in the list of maps.

  • Visible: If a map is not visible, then it will still run, it will just not be included in the transformed record. This is useful if you’re doing a multi-step transformation and you don’t want to output your intermediates. In general, just make everything visible, because it’s simpler.

  • Source Table: Rename maps can create columns from either the raw or transformed record. Usually you want to select “Raw” here, but when you are either copying a field already in the transformed record or using the output of a function transform (see Function Transform below), you need to select “Transformed”.

  • Column Type: Explained below. Usually you want “Metadata”.

  • Input Column: The name of the column you want to take as input, either from the raw or transformed record depending on your selection in the Source Table field.

  • Output Column: The name of the column you want to create in the transformed record. Generally, this should be a relatively short upper case string. Exceptions are that you should call the subject ID “subjectId” and the visit ID “visitId”. I am aware that this inconsistency sucks, but due to historical inertia, we’re stuck with it for now.

Most of this should sound pretty straightforward. Source Table might be a bit confusing. But Column Type is definitely confusing. Let me explain what all the values mean:

  • Metadata: 99% of your rename transforms will be Metadata. Originally etadata was supposed to mean “this column is in the primary key of the output domain”, in which case it could be used to distinguish between updated records and new records, which could eventually be exposed to users in CMR. However, due to limitations in the Attribute column type (described below), I use Metadata to mean “anything that doesn’t need to be one of the other column types”, which is the vast majority.

  • Observation: Making everything Metadata works for going from 1 raw record to 1 transformed record. But sometimes you need to go from 1 raw record to N transformed records. Like imagine if HEIGHT and WEIGHT come in the same raw record and we want them to produce 2 transformed records. This is ultimately accomplished by a Pivot Transform (described below), but the unpivot operation needs to know which values get turned into new rows. So in that case, HEIGHT and WEIGHT would need to be Observations.

  • Attribute: An Attribute is a child of an Observation. Extending the above column, imagine HEIGHT_UNITS and WEIGHT_UNITS columns. You want HEIGHT_UNITS in the unpivoted HEIGHT row, and WEIGHT_UNITS in the unpivoted WEIGHT row. To do that, label them as Attributes and fill in the “Belongs To” field with the name of the parent Observation.

    (In theory you could use Observations and Attributes rather than Metadata for columns that are not in the primary key, but this winds up being confusing because there’s often to clear Observation in a record, and also there is no such thing as a “global” Attribute that applies to any Observation or no Observation, which forces you to always use Metadata columns for that purpose when unpivoting. For these reasons, I just use Metadata as much as possible. Less confusing, more consistent.)

  • SubjectId: Due to some internal implementation details of the mapping engine, SDW needs to know which column contains the subject ID when you do a pivot transform (described below). For an output domain that contains a pivot transform, set the column type of the column containing the subject ID to “SubjectId”. For output domains that do not contain a pivot transform, use a column type of “Metadata”, like normal; otherwise, you may see performance issues, especailly with large amounts of data. (And remember, as mentioned above, set the Output Column to “subjectId”.)

  • Key and Value: An output domain often contains multiple different types of data. For example, VS (vital signs) will have rows for different measurements - WEIGHT, HEIGHT, SYSBP, DIABP, TEMP, etc. Based on SDTM, we generally put the result type in a column called TESTCD and the result value in a column called ORRES (if original units) or STRESC (if converted to standardized units). This is useful for downstream applications because if they just want to get the data for WEIGHT, they can query for “VS.WEIGHT” rather than “VS”. We call this a “subdomain” (VS is the domain, WEIGHT is the subdomain), and that feature is enabled by the Key and Value column types. TESTCD must be a Key type to support subdomain querying, and then ORRES/STRESC should be a Value column. (Actually I think you don’t even need to use the Value type, you could just make it Metadata… but I’m not 100% sure. Key is definitely required for this feature, though). See also the Subdomains section below.

Okay, so that was all pretty confusing. But hopefully you understand why we need to have column types. The take-home message is… put the subject ID in a SubjectId column, put nearly everything else in a Metadata column, and deviate from that only when you absolutely need to.

Join Transform

Joins allow you to pull in data from other input domains. Each join transform retrieves the value from a single column in another input domain, and puts that value in a new column in the transformed record.

A common use of a join is to get the reference start date (RFSTDAT) of a subject. This is the date when a subject started taking a drug. It is stored in one place, but it is used in nearly every output domain to calculate “study day” values - like AE.STDY is “number of days between when the subject started taking the drug and when the adverse event occurred”.

Let’s look at all the information you need to specify to do a join:

  • Name, Mapset Name, Enabled, Visible, Source Table, Column Type, Output Column: Exactly the same as for a Rename Transform (see above).

  • Selected Domain: The input domain you want to pull a value from.

  • Selected Column: The column in the Selected Domain containing the value you want to pull.

  • Join Conditions: This is the tricky part. Basically it’s the WHERE clause in a SQL query. Press [+ Add] for each condition.

    Target Col is the name of the column in the Selected Domain that you want to use in a condition. Target Col must always be filled out. Then, fill out either Source Col or Literal Value based on your desired condition.

    In Source Col, put the name of a column in the current input domain whose value must be equal to the value of the Target Col. Often you want Source Col and Target Col to be the same, like in ODM when joining within the same subject, “SubjectKey” will be both Source Col and Target Col. If you need to use a processed value (a column from “Map Results” rather than “Source Data Preview”) then switch the Source Table From Raw to Transformed. Try to avoid this though, because it’s really slow.

    Literal Value lets you apply a filter to Target Col. Like if there is a Target Col called Visit and you only want the first visit, you would put something like “V1” in the Literal Value field.

Overall, it’s quite similar to joins in SQL.

The tricky part is, it’s really difficult to debug when it doesn’t work. Usually it’s caused by some typo somewhere, but there is no way for it to figure out what the typo is.

Another challenge is the cardinality of the input domains you’re joining. In the simple case, your join conditions results in one value. If it results in no value, that’s also fine, it just becomes null. But what if it results in two or more values? Then SDW will create two separate transformed records, one with each value. Maybe this is what you want sometimes, but not usually. When this happens, it can be both difficult to notice it’s happening (you have to count the rows and notice the difference) and difficult to debug (there is no indication of which join map caused the extra rows to appear). Good luck!

ODM/eCOA Joins

For ODM/eCOA data, we know what the primary key is. It’s all the OIDs and RepeatKeys. Therefore, when doing a join to an ODM/eCOA domain, we know what join keys we need to specify to avoid the cardinality problem mentioned above. These should be included in your join conditions when possible:

  • SubjectKey
  • StudyEventOID
  • FormOID

(StudyOID and ItemGroupOID are not needed because there are no cross-study joins and ItemGroupOID is the same as the input domain name.)

Additionally, for repeating visits/forms/itemgroups, include these in your join conditions when possible:

  • StudyEventRepeatKey
  • FormRepeatKey
  • ItemGroupRepeatKey

For non-repeating ones, those columns won’t exist.

This matters not just for correctness, but for performance. Due to limitations in the Apache Phoenix query planner, sometimes it will run out of memory when doing a join. The odds of this happening can be reduced if you add all the join conditions above, and if for ones where the value is a constant you use “Literal Value” rather than “Source Col” to specify the condition.

It can be tempting to say “Oh, I know that I don’t need to include X in my join conditions because it never affects the outcome of the join, so I’ll just skip it.” Sometimes you can get away with that, and even sometimes you are forced to do that because you just don’t have all the values available to you. But sometimes you can’t, either due to performance reason mentioned in the previous paragraph or because your assumption might not be correct.

Explicit is better than implicit!

Pivot Transform

A pivot transform lets you swap between this table:

WEIGHT WEIGHT_UNITS HEIGHT HEIGHT_UNITS DAT
100 lb 72 in 2017-01-02
110 lb 74 in 2017-04-05

and this table:

TESTCD ORRES ORRESU DAT
WEIGHT 100 lb 2017-01-02
HEIGHT 72 in 2017-01-02
WEIGHT 110 lb 2017-04-05
HEIGHT 74 in 2017-04-05

In this example, the forward operation is called “depivot” (other systems call it “unpivot”, but SDW is special) and the backwawards operation is called “pivot”. Pivoting is very rare, but unpivoting is common in vital signs forms.

(Also about the above example… WEIGHT and HEIGHT are observations, WEIGHT_UNITS and HEIGHT_UNITS are attributes, and DAT is metadata.)

Here are the details of defining a pivot transform:

  • Name, Mapset Name, Enabled: Exactly the same as for a Rename Transform (see above).

  • Action: Depivot or Pivot, based on the direction you want to go.

    When “Depivot” is selected:

    • Value Column Rename: The name of the column containing the value of the observations. In the above example, it would be “ORRES”.

    • Transform: Add an entry for the name of the key column. In the above example, there would be one entry for “TESTCD”. I have never needed more than one entry here.

    See also the Subdomains section below.

    When “Pivot” is selected: …I have never done this.

Function Transform

A function transform is a block of JavaScript code that operates on the transformed record.

The Name, Mapset Name, and Enabled fields are the same as they are in all the other maps.

Function Body contains code. It’s kind of confusing. Imagine a JavaScript function like this:

function myFunction(r) {
    r.NEW_FIELD = 'value';
    return r;
}

That is a function that takes a single argument r, adds a column to it, and returns it.

In SDW, you would write that as:

r.NEW_FIELD = 'value';
return r;

Basically, the first and last lines have been removed, and you just have to write the body of the function. r always contained the transformed record.

THIS IS VERY IMPORTANT! If you create a function transform like that, you will notice that no column called NEW_FIELD appears in the output. That is because, just given the above code, there is no way to know the column type. To set the column type, create a rename transform that runs after the function transform and creates a new column called NEW_FIELD based on NEW_FIELD in the transformed table.

Creating a new column with some constant text, as described above, is a common application of function transforms. Others include date formatting, unit conversion, and study day calculation. If you use the programmatic API, there is a large library of pre-written functions you can use. If you don’t want to use the programmatic API, you could copy and paste the code from its function transforms into the UI.

Another thing to remember is to always return string values (or null), not numbers or any other JavaScript data types. Example:

r.visitId = String(parseInt(r.visitId, 10));
return r;

The above code converts visitId to an integer, and then converts the integer to a string. So for an input like “10.1”, the output would be “10” (as a string, not a number). If something besides a string or null is in the transformed record, downstream applications might break. Unfortuantely for you, there is nothing that checks or validates this, you just have to be careful.

Filtering

Filtering is done in function transforms, rather than as a dedicated map type. To filter, return null for any record you want to remove from the output. Example:

if (r.visitId === 'V1') {
    return r;
}
return null;

That will filter out any records where the visitId is something besides “V1”.

Code List Transform

Code list transforms allow you to apply a dictionary lookup, like going from M/F to Male/Female. I recommend you don’t use them, though, because you have no control over handling the case where a value is not in the dictionary. Like if the input was M/F/? given the M/F only dictionary, the ? would get converted to (?). Maybe that’s what you want, but probably it’s not. I just write a function transform to do it, then I have full control. This is built into the dictionary feature of the programmatic API.

Subdomains

Subdomains are created when you have either key/value rename transforms or a depivot transform. The purpose of this is to enable querying a subset of a domain, like “VS.STRESN” means “numeric results of all vital signs” while VS.WEIGHT.STRESN means “numeric result of only the weight measurements”.

For vital signs, this is generally pretty simple. See the Standard Domains and Columns table for some standard VS subdomains. But for lab data (LB), it’s usually difficult. The problem is, lab data usually comes associated with an ugly test code that nobody can read and interpret, like RCT1 for bilirubin. Nobody knows that RCT1 is bilirubin. In SDTM, the test code is more easily interpretable: BILI. That’s what we want. But there are some problems:

  1. Often we are not provided a dictionary to convert between ugly test codes and SDTM test codes. Even for internal Covance lab data, I still have not been able to get a full list.

  2. Test codes change across studies. Different sponsors might use different codes, in which case you need to track down another translation table.

  3. There is not a one-to-one relationship between internal lab test codes and SDTM test codes. In fact, it is many-to-one. The problem is, in SDTM v3.1.2, the primary key of the LB domain is (STUDYID, USUBJID, LBTESTCD, LBSPEC, VISITNUM, LBTPTREF, LBTPTNUM). The most common issue is multiple tests having the same test code but a different specimen. Like if you measure glucose in blood and in urine, they are not directly comparable, but they both have a test code of GLUC so if we mapped them to the LB.GLUC subdomain it would return both. We generally solve this by concatenating the specimen and the test code, making it them LB.GLUC_URINE or LB.GLUC_BLOOD. Those are ugly, but at least they don’t mix unrelated data.

Please reach out to me Jeremy.Scheff@covance.com if you have issues with this, particularly related to obtaining test code dictionaries and coming up with unique test codes.

Error Handling

The Mapping Errors panel on the mapping page gives a list of errors that occurred when applying the maps for the Map Results preview table, such as syntax errors in function transforms or rename transforms referencing columns that don’t exist.

Ultimately, as long as you get the results you want, it doesn’t matter what errors are shown. However, if possible, it is nice to keep the error list empty, because otherwise you might not notice when you introduce an error if you’re used to ignoring a big list of benign errors.

Don’t Repeat Yourself

“Don’t repeat yourself” (DRY) is a common programming mantra. Basically it means… try not to write the same thing twice, becuase that’s extra work for no good reason, and it makes it harder to maintain in the future because now any change you make has to happen in sync in two places.

The best way to stay DRY is to use the programmatic API described below. But even in the UI, there are some features that help a little.

You can clone an output domain by clicking [+ EXISTING] in the Output Domains panel of the mapping screen. This may be useful if a study has many similar forms or if there are two very similar studies.

The downside is, even when things are very similar, you will often find yourself having to go through every map and rename columns.

Similarly, individual maps can be cloned by clicking [+ EXISTING] in the Maps panel, but again they may require some customization.

There are still some unsolved problems related to DRY in the UI:

  • No way to bulk edit maps. Like to rename all output columns in this study named X to Y, currently you’d have to do them all individually. Or if you found a typo in your BMI calculation function that affects every study, you’d have to edit the code individually for each study.

  • No way to import or reference a group of maps (except an entire output domain), such as a set of maps used to calculate a common derived variable such as “study day” or “BMI”.

  • No way to specify similar but varying maps. Like if there are input domains CM1, CM2, CM3, etc. all with the same columns but prefixed with the name of the input domain (like CM1TRT, CM2TRT, CM3TRT), using the UI even if you copy the output domain you will have to go and manually rename every column.

  • No way to reuse parts of functions. Like if you write a function to convert date formats and you use that in a few different places (calculating different study days, calculating durations of AEs, formatting dates for display, etc.), there is no way to share that common function across different function transforms.

Eventually the UI will provide this functionality. If you want it today, you can use the programmatic API.

Standard Domains and Columns

In some ideal world, SDW mapping would be the same as SDTM mapping. There are two problems with this. Sometimes people want to look at data in CMR that doesn’t exist in SDTM, in which case you’d have to add extra columns or domains. And in general, documentation on SDTM mapping is not available when we are mapping in SDW.

Currently, the process to figure out what to map in SDW has three components:

  1. Map standard columns in standard domains that we expect to appear in nearly every study. See the table below. These should be similar to the SDTM standard.

  2. Map other columns that seem to go along with the standard domains in this specific study. In an ideal world, you would map them as close to the SDTM standard as possible, but sometimes that is difficult.

  3. Map other columns and domains requested by users (sometimes this is nothing, sometimes it’s a few things, sometimes it’s a lot of stuff).

CMR Datamart

Currently, there are two applications that use data from SDW - CMR (Centralized Medical Review) and CSM (Centralized Statistical Monitoring). These applications do not directly read data from SDW. Instead, they read it from a data mart called cmr-datamart. Ultimately, data mapped in SDW has to be loaded into the data mart and then verified in CMR or CSM to confirm that everything actually works. In practice, 99% of the time you can just do the mapping in SDW and everything downstream will work fine.

Programmatic API
